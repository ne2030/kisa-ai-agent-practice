Large Language Model

LLM, 글자 그대로 풀어보면

  • Large — 모델 파라미터 수십억~수천억 개 · 학습 데이터는 수조 토큰 규모
  • Language — 자연어 (사람이 쓰는 일상 언어). 기계어(Python·Java·assembly 같은 프로그래밍 언어)와 구분되는, 사람의 글
  • Model — 입력→출력 함수. 그 함수의 구조가 Transformer 아키텍처
토큰 단어를 잘게 쪼갠 단위 — 예: "안녕하세요" → "안녕", "하세요". 자주 등장하는 패턴은 한 토큰, 드문 단어·표현은 여러 토큰으로 쪼개진다 (정확한 분할은 모델·토크나이저마다 다름). LLM은 입출력을 모두 토큰 단위로 처리한다.

for call in response.tool_calls:
    # 1. validate — 등록된 tool인가?
    if call.name not in HANDLERS:
        raise UnknownToolError(call.name)

    # 2. call — host가 실제 함수 실행

Planning 패턴

다음 행동을 정하는 reasoning의 종류

↳ Lilian Weng (2023) Planning component 안의 sub-pattern들. 이 강의는 가장 단순한 ReAct.

Inline (per-step)

  • 매 decide step에서 reasoning + 결정을 한 LLM 호출에
  • 별도 planner 함수 없음 — 코드에 안 보임
  • 예: ReAct ← 이 강의 데모, CoT
  • 단순 · 적응적

Pre-computed

  • 별도 LLM call이 전체 plan 생성
  • 실행 loop는 plan을 step별로 소비
  • 예: Plan-and-Execute, BabyAGI
  • 다단계 task · replan 가능 (Section 5/6)

ReAct는 한 LLM 호출 안에서 reasoning + action 결정을 함께 하는 self-reflection 계열 — 가장 단순해서 이 강의의 데모로 사용.

기본 구현은 약 40줄 안에

import os
from google import genai

# 1) Tool registry
def search_db(query: str) -> str:
    return f"results for: {query}"

TOOLS = [{
    "name": "search_db",
    "description": "사내 DB 검색",
    "parameters": {
        "type": "object",
        "properties": {"query": {"type": "string"}},
        "required": ["query"],
    },
}]
HANDLERS = {"search_db": search_db}

# 2) ReAct loop
def react_loop(user_input: str, max_steps: int = 10):
    client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])
    messages = [{"role": "user", "content": user_input}]

    for _ in range(max_steps):
        resp = client.models.generate_content(
            model="gemini-2.5-flash",
            contents=messages,
            config={"tools": TOOLS},
        )

        if not resp.tool_calls:               # stop
            return resp.text

        for call in resp.tool_calls:          # act + observe
            if call.name not in HANDLERS:
                raise ValueError(f"unknown: {call.name}")
            result = HANDLERS[call.name](**call.arguments)
            messages.append({"role": "tool",
                             "name": call.name,
                             "content": str(result)})

    raise RuntimeError("max_steps exceeded")

RAG 평가

검색과 생성을 분리해서 측정

Retrieval — 검색 품질

  • Precision@k — top-k 결과 중 관련 문서 비율

  • k 키우면 recall↑ precision↓ 트레이드오프

Generation — 응답 품질

  • Faithfulness — 답변이 검색 결과에 충실한가 (외부 지식 X)
  • Answer relevance — 답변이 질문에 부합하는가
  • Context utilization — 검색 결과를 실제로 사용했는가

왜 분리하나 — 답변이 틀렸을 때 검색이 잘못된 건지, 생성이 잘못된 건지 분리해야 어느 layer를 고칠지 보인다.

전문화
전문 분야 + 전용 system prompt + 전용 tool set. tool 50개를 한 LLM이 들고 있으면 적절한 도구를 못 고른다 — 컨텍스트 폭발, 역할 혼란, trace 복잡.
②
컨텍스트 한계
큰 작업은 short-term memory에 다 안 들어간다. 분할하면 각자 자기 작업에 필요한 것만 들고 간다. 모델 혼란이 줄고, 토큰 비용도 줄어든다.
③
병렬화
동시 실행으로 latency 단축 또는 throughput 증가. 두 형태:
  • 작업 분해 다른 일을 동시에 — 한 요청을 검색·검증·요약으로 분해해 동시 실행 (latency ↓)
  • 데이터 분산 같은 일을 N번 — 동일 에이전트를 데이터 슬라이스별로 fan-out (예: 뉴스 수집을 국가별 N개) (throughput ↑)

①② = "한 LLM으로 안 됨" (인지 한계). ③ = "직렬은 느림" (시간 한계). 모놀리스 → 마이크로서비스 진화의 동일한 압력.

비용 효율화 ① context 줄이기

매 LLM 호출의 input 길이가 곧 비용이다

agent는 한 호출마다 system instruction, tool 정의, 대화 history, 검색 결과, 이전 tool 응답을 모두 합쳐 input으로 보낸다. 이 input을 작게 유지하는 게 가장 직접적인 비용 절감.

나쁜 패턴

  • Tool 응답을 통째로 투입 — 외부 API JSON 30KB를 그대로 다음 호출 input에 붙임. 정작 모델이 쓰는 필드는 5~10개뿐.
  • 대화 history 무한 누적 — 사용자와 주고받은 모든 turn을 매 호출마다 다시 첨부. 호출이 길어질수록 input이 monotonic하게 커짐.
  • 검색 결과 과도하게 열기 — 검색 증강(RAG)에서 관련도 낮은 chunk까지 넉넉히 가져와 다 넣음.

좋은 패턴

  • 필요한 필드만 추출 / 짧게 요약 — tool 응답 30KB 대신 5~10개 필드만 뽑아 800 tokens 이내로 줄여 전달.
  • 오래된 history는 요약으로 압축 — 최근 N턴은 그대로, 그 이전 대화는 한두 단락 요약(summary)으로 대체.
  • 상한선을 정책으로 강제 — feature별 input · output 토큰 상한, 검색 chunk 개수 상한을 미리 설정.

Prompt Injection 방어

Context 경계 + Action 경계

injection은 LLM 자체의 alignment로 막을 수 없다. 모델 바깥에 두 종류의 경계를 둬야 한다 — 모델이 *보는 것*(context)을 통제하고, 모델이 *시킬 수 있는 것*(action)을 통제.

Context 경계 — LLM이 보는 입력 통제

  • untrusted data 분리 — system instruction은 위, 사용자 입력 · 검색 결과는 명확한 구분자(<user_input>, <retrieved>)로 감싸 모델이 "이건 데이터지 지시가 아니다"라고 알 수 있게
  • 검색 문서 sanitizer — RAG 문서에서 hidden text · 안 보이는 unicode · metadata 제거 후 context 투입
  • retrieval 신뢰도 등급 — 출처별로 trust level 부여, 외부 웹 문서는 read-only로 격리
  • 대화 누적(conversation-level) 탐지 — 한 번에 위험 지시를 안 하고 한 턴씩 조금씩 경계를 밀어붙이는 점진적 공격은, 한 턴이 아니라 대화 전체 패턴으로 모니터링해야 잡힘

Action 경계 — LLM의 결정이 실행되기 전 검증

  • tool allowlist + arg schema — agent가 부를 수 있는 tool 화이트리스트, 각 인자가 schema에 맞는지 검증, tenant ACL로 사용자별 권한 분리
  • read · write 분리 — read tool(검색·조회)은 자유 호출, write tool(메일 발송 · DB write · 결제)은 사람 승인 또는 dry-run 우선
  • canary token — system prompt나 sensitive context에 고유 token을 박아두고, 응답에 그 token이 새 나오면 누출 알람
  • synthetic injection 회귀 테스트 — 알려진 injection payload 데이터셋을 매 배포 전 돌려 regression 체크, 새 우회 패턴은 dataset에 추가
