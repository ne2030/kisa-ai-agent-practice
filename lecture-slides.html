Large Language Model

LLM, 글자 그대로 풀어보면

  • Large — 모델 파라미터 수십억~수천억 개 · 학습 데이터는 수조 토큰 규모
  • Language — 자연어 (사람이 쓰는 일상 언어). 기계어(Python·Java·assembly 같은 프로그래밍 언어)와 구분되는, 사람의 글
  • Model — 입력→출력 함수. 그 함수의 구조가 Transformer 아키텍처
토큰 단어를 잘게 쪼갠 단위 — 예: "안녕하세요" → "안녕", "하세요". 자주 등장하는 패턴은 한 토큰, 드문 단어·표현은 여러 토큰으로 쪼개진다 (정확한 분할은 모델·토크나이저마다 다름). LLM은 입출력을 모두 토큰 단위로 처리한다.

입력 이해

모든 토큰이 서로를 참조해 의미를 잡는다

정리

Tool 정의

JSON schema로 LLM에 알려준다 

# tool registry — 인터페이스 명세 + 실제 함수 매핑
def search_db(query: str) -> str:
    return f"results for: {query}"

TOOLS = [{
    "name": "search_db",
    "description": "사내 DB에서 키워드 검색",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {"type": "string"}
        },
        "required": ["query"]
    }
}]
HANDLERS = {"search_db": search_db}  # 이름 → 함수

→ LLM은 이 schema를 system context처럼 받아서 "어떤 도구가 있고, 어떤 인자를 받는지" 안다.

Host의 실행

validate → call → observe 

for call in response.tool_calls:
    # 1. validate — 등록된 tool인가?
    if call.name not in HANDLERS:
        raise UnknownToolError(call.name)

    # 2. call — host가 실제 함수 실행
    result = HANDLERS[call.name](**call.arguments)

    # 3. observe — 결과를 messages에 추가
    messages.append({
        "role": "tool",
        "name": call.name,
        "content": str(result)
    })

→ 다음 LLM call은 이 결과까지 본 messages 배열을 다시 받음 — loop continues.

Planning 패턴

다음 행동을 정하는 reasoning의 종류

↳ Lilian Weng (2023) Planning component 안의 sub-pattern들. 이 강의는 가장 단순한 ReAct.

Inline (per-step)

  • 매 decide step에서 reasoning + 결정을 한 LLM 호출에
  • 별도 planner 함수 없음 — 코드에 안 보임
  • 예: ReAct ← 이 강의 데모, CoT
  • 단순 · 적응적

Pre-computed

  • 별도 LLM call이 전체 plan 생성
  • 실행 loop는 plan을 step별로 소비
  • 예: Plan-and-Execute, BabyAGI
  • 다단계 task · replan 가능 (Section 5/6)

ReAct는 한 LLM 호출 안에서 reasoning + action 결정을 함께 하는 self-reflection 계열 — 가장 단순해서 이 강의의 데모로 사용.

Demo · ReAct loop 전체

기본 구현은 약 40줄 안에

import os
from google import genai

# 1) Tool registry
def search_db(query: str) -> str:
    return f"results for: {query}"

TOOLS = [{
    "name": "search_db",
    "description": "사내 DB 검색",
    "parameters": {
        "type": "object",
        "properties": {"query": {"type": "string"}},
        "required": ["query"],
    },
}]
HANDLERS = {"search_db": search_db}

# 2) ReAct loop
def react_loop(user_input: str, max_steps: int = 10):
    client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])
    messages = [{"role": "user", "content": user_input}]

    for _ in range(max_steps):
        resp = client.models.generate_content(
            model="gemini-2.5-flash",
            contents=messages,
            config={"tools": TOOLS},
        )

        if not resp.tool_calls:               # stop
            return resp.text

        for call in resp.tool_calls:          # act + observe
            if call.name not in HANDLERS:
                raise ValueError(f"unknown: {call.name}")
            result = HANDLERS[call.name](**call.arguments)
            messages.append({"role": "tool",
                             "name": call.name,
                             "content": str(result)})

    raise RuntimeError("max_steps exceeded")

[trace] react_loop("production logs 분석")  · 4.2s · $0.012

평가가 없으면

무엇을 측정하나 — Rubric

정성적 차원을 1-5점 척도로 명시

  • correctness — 사실 정확성. reference가 있으면 직접 비교
  • helpfulness — 사용자 의도 부합도. 정답이 여럿일 수 있는 상황에서
  • safety — 유해·민감·차별 여부
  • 채점 방법 — 위 차원은 자유로운 응답의 의미를 평가해야 하므로 코드로 못 잡음 → LLM-as-Judge가 1-5점 부여 (3단계에서 자세히)

Rubric을 명시하는 행위 자체가 프로덕트 요구사항을 정리하는 작업 — 평가의 강력한 부수효과.
(출력 형식·schema 준수 같은 결정적 체크는 Rubric이 아니라 2단계 Code-based assertion에서 다룸)

무엇을 측정하나 — Metric

정량적인 것은 코드로 직접

  • accuracy — 정답 일치율 (분류·추출처럼 정답이 있는 task)

  • tool call count — 한 요청당 LLM·tool 호출 수. 효율성 지표

Section 04●Metric
43 / 96
RAG 평가

검색과 생성을 분리해서 측정

Retrieval — 검색 품질

  • Precision@k — top-k 결과 중 관련 문서 비율
  • Recall@k — 전체 관련 문서 중 검색된 비율
  • k 키우면 recall↑ precision↓ 트레이드오프

Generation — 응답 품질

  • Faithfulness — 답변이 검색 결과에 충실한가 (외부 지식 X)
  • Answer relevance — 답변이 질문에 부합하는가
  • Context utilization — 검색 결과를 실제로 사용했는가

왜 분리하나 — 답변이 틀렸을 때 검색이 잘못된 건지, 생성이 잘못된 건지 분리해야 어느 layer를 고칠지 보인다.

채점 진화 — 2단계

Code-based assertion — 결정적 룰로 회귀 차단

  • must_call_tools — 기대한 tool들이 호출됐는가 (예: oauth 질문엔 search_db 호출 필수)
  • must_mention / must_not_say — 답변에 특정 표현 포함/제외 (예: "I don't know" 회피 답변 금지)
  • schema validation — 출력 JSON 형식 준수
  • arg validation — tool 호출 인자가 schema에 맞는가
  • 장점 — deterministic, fast, cheap. CI에서 즉시 실행
  • 한계 — 의미 검증은 못함 → 다음 단계 LLM-as-Judge로 보완

Judge prompt 예시

평가용 system prompt + scoring rubric

# judge LLM에게 보내는 prompt
SYSTEM:
You are evaluating an AI agent's response.

Task:           {task}
Agent's Answer: {answer}
Reference:      {reference}    # 있으면 비교 기준

Score on 1-5 scale:
  - correctness  — answer matches reference?
  - helpfulness  — answers user's intent?
  - safety       — any harmful content?

Return JSON:
{
  "correctness": 1-5,
  "helpfulness": 1-5,
  "safety":      1-5,
  "reasoning":   "brief explanation"
}

→ 이 prompt를 다른 LLM에 보내서 JSON 점수 회수. 평균이 threshold (예: 4.0) 이상이면 통과로 처리.

두 종류의 평가

어떻게 나아갈 것인가 / 어떻게 유지할 것인가

Capability — 역량 평가

  • 질문 — "이 에이전트가 무엇을 잘 하는가?"
  • 어려운 과제로 시작 — 통과율 낮아도 OK
  • 개선되며 통과율이 100%에 가까워지면 졸업 (새 어려운 과제로 이동)
  • 예: 새 도메인 과제, 어려운 사례 모음

Regression — 회귀 평가

  • 질문 — "이전에 잘 하던 걸 여전히 잘 하는가?"
  • 항상 ~100% 통과율 유지가 목표
  • 점수 하락 = 어딘가 깨짐 → 즉시 수정
  • 예: 이미 풀던 과제, 과거 실패 → 재발 방지

Anthropic — "Hill-climb on capability evals, but run regression evals to catch breakage elsewhere." 두 축이 함께 굴러가야 안전.

Token 감각

1M tokens는 생각보다 금방 쌓인다

  • 1,000 tokens × 1,000 calls = 1M tokens
  • 10,000 tokens × 100 calls = 1M tokens
  • 100,000 tokens × 10 calls = 1M tokens
  • RAG context, tool result, conversation history가 붙으면 한 call이 10K tokens를 넘기 쉽다
  • agent가 사용자 요청 하나에서 LLM을 5번 부르면, 1,000명 요청이 아니라 200명 요청만으로도 1M-call 규모에 가까워진다

각 layer의 책임
